Overview

The problem and solution

The challenge: Domain experts accumulate vast PDF archives — research papers, technical manuals, regulatory documents — but finding specific information buried in these documents is time-consuming and error-prone.

Traditional keyword search fails when you don't know the exact terminology. You need a system that understands *meaning*, not just matches words.

The solution: A RAG (Retrieval Augmented Generation) system that converts documents into semantic embeddings, stores them in a vector database, and uses an LLM to generate accurate answers grounded in the source material — with citations.

This project demonstrates the full pipeline: from PDF ingestion to conversational Q&A, with emphasis on production patterns like auto-indexing, proper chunking strategies, and citation tracking.

Architecture

How the system works

Pipeline Breakdown

Document Ingestion

PDFs are loaded with pdfplumber, preserving page numbers and metadata. Text is extracted while handling complex layouts and tables.

Smart Chunking

Documents are split into overlapping chunks using LangChain's recursive splitter. Overlap ensures context isn't lost at chunk boundaries.

Embedding Generation

Each chunk is converted to a 1024-dimensional vector using Voyage AI's retrieval-optimized embeddings.

Vector Storage

Vectors are stored in Pinecone with metadata (source file, page number). Cloud-hosted for persistence and scalability.

Semantic Retrieval

User queries are embedded and matched against stored vectors using cosine similarity. Top-k most relevant chunks are retrieved.

Answer Generation

Claude receives the query + retrieved context, generating an answer grounded in the source material with proper citations.

Key Decisions

The why behind the choices

Why Voyage AI over OpenAI embeddings?

Three reasons: Anthropic partnership (official recommendation), free tier (200M tokens — enough for serious development), and retrieval optimization (embeddings specifically tuned for search, not just similarity).

OpenAI's embeddings are general-purpose. Voyage's are built for RAG.

Why Pinecone over local vector stores like Chroma?

Persistence without infrastructure. Chroma requires managing a database process. Pinecone is cloud-hosted — vectors persist across restarts, sessions, and machines.

For a portfolio project demonstrating production patterns, showing cloud-native architecture is more valuable than local-only solutions.

Why pdfplumber over PyPDF?

Complex layouts. Technical documents often have multi-column text, tables, and figures. pdfplumber handles these better than PyPDF, which often garbles text order.

The tradeoff is speed (pdfplumber is slower), but accuracy matters more for RAG — garbage in, garbage out.

Why include a folder watcher?

Production pattern. Real document systems don't require manual indexing commands. The folder watcher demonstrates event-driven architecture: drop a PDF, it's automatically indexed.

This is the foundation for integrations like Google Drive sync or Dropbox webhooks.
