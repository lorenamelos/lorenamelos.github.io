AI Job Hunting Agent

Started Jan 2026 11 phases completed ~430 tests passing

An autonomous system that discovers job postings from multiple sources, scores them against my profile using LLM-powered matching, generates tailored resumes, and delivers weekly digests via email. Built with defensive architecture principles — state machines, graceful failure, and full observability.

The problem

Job hunting is a manual grind

Searching for AI/ML engineering roles means checking multiple job boards daily, reading through dozens of job descriptions to assess fit, tailoring resumes for each application, and tracking what you've applied to. It's repetitive, time-consuming, and error-prone.

I wanted to automate the entire funnel — from discovery to resume generation — while maintaining human oversight over the final application decision. The system runs autonomously, but I control what gets submitted.

System architecture

The pipeline at a glance

The system is a multi-stage pipeline orchestrated by GitHub Actions. Each stage has a single responsibility, with clear boundaries and failure isolation.

Discovery Dedup Fetch Filter Match (LLM) Tailor (LLM) Resume Gen Email

High-level data flow

The pipeline splits into two workflows: a daily run that discovers and scores new jobs, and a weekly run that generates tailored resumes and sends the digest. This separation prevents cross-VM file issues and keeps costs predictable.

State management

Jobs as a state machine

Every job in the database has a state that tracks its progress through the pipeline. This enables graceful failure recovery — if the matching stage fails mid-run, we can resume from exactly where we stopped.

Job state transitions

The state machine ensures idempotency: running the same stage twice on the same job is a no-op. This is critical for CI/CD reliability — if GitHub Actions retries a failed run, we don't reprocess jobs or send duplicate emails.

Anti-hallucination

LLM with guardrails

The biggest risk with LLM-generated resumes is hallucination — the model inventing skills or experiences I don't have. Phase 10 introduced a tool-calling architecture that makes this structurally impossible.

Resume tailoring with validation

The LLM's only job is to select which pre-written bullet points best match the job description. It returns a list of IDs, which are then validated against the master resume. If an ID doesn't exist, the validator rejects it. The generator only uses validated IDs to build the final document.

Key decisions

Trade-offs and rationale

  • Separate daily/weekly workflows — GitHub Actions VMs are ephemeral. Running discovery and email in the same workflow meant generated resumes were lost between stages. Splitting them with Supabase as the handoff point solved this.
  • Fingerprint-based deduplication — The same job appears on multiple boards (RemoteOK, WeWorkRemotely, company ATS). A normalized fingerprint (company + title + location hash) ensures each job is processed once.
  • 70% match threshold — After testing, jobs below 70% match score rarely led to interviews. The threshold balances recall (don't miss good jobs) with precision (don't waste time on poor fits).
  • No selection phase — Originally planned a manual review step before resume generation. In practice, the matching phase is accurate enough that all matched jobs go straight to resume generation.
  • Tool calling for anti-hallucination — Free-form LLM output required complex parsing and still hallucinated. Tool calling with ID-only returns made validation trivial and hallucination structurally impossible.
  • Resumes not in git — Generated .docx files contain personal info and are job-specific. They go directly to email, not tracked in the repository.
